# Техническое задание: Калькулятор калорий по фото (Photo Calorie Calculator)

## 1. Название и роль проекта в системе

**Photo Calorie Calculator** — мобильное приложение для автоматического распознавания еды по фотографии, оценки её объема и расчета пищевой ценности (калорийности и БЖУ). 

Проект выступает в роли умного помощника для трекинга питания, который переносит вычислительную нагрузку по распознаванию образов (с использованием нейросети YOLOv8) на серверную часть, оставляя клиентское приложение легковесным. Основная роль системы — максимально упростить процесс дневника питания, заменив ручной ввод и взвешивание еды на один клик камеры.

Тип проекта: клиент-серверное мобильное приложение с ML-компонентом на бэкенде.

## 2. Решаемая задача

В классических приложениях для подсчета калорий (дневниках питания) пользователю приходится вручную искать продукт в базе данных, а затем самостоятельно указывать его вес, что требует наличия кухонных весов или оценки "на глаз". Это создает высокий порог входа и приводит к тому, что пользователи быстро забрасывают трекинг из-за сложности процесса.

**До приложения:** Пользователь готовит или покупает еду, взвешивает каждый ингредиент, ищет его в приложении, записывает граммы. Если весов нет — угадывает вес с высокой погрешностью.

**После приложения:** Пользователь делает фотографию тарелки. Нейросеть распознает тип продукта и оценивает его примерный объем. На основе заранее заложенной плотности продукта система автоматически высчитывает примерный вес и БЖУ. Результат мгновенно сохраняется в историю. Трекинг занимает несколько секунд.

## 3. Главная гипотеза

Если предоставить пользователям возможность оценивать калорийность и БЖУ блюда по одной фотографии с приемлемой точностью, то они будут вести дневник питания гораздо регулярнее благодаря радикальному снижению когнитивной и административной нагрузки (отсутствие ручного ввода и необходимости взвешивать).

## 4. Пользователи и бизнес-сценарии

**Основные пользователи:**
- Люди, следящие за своим питанием и фигурой (похудение, набор массы);
- Спортсмены, нуждающиеся в быстром трекинге макронутриентов;
- Пользователи, которым лень вести классические дневники питания из-за необходимости ручного взвешивания.

**Сценарии использования:**

*Сценарий 1. Быстрый трекинг приема пищи.*
Пользователь открывает приложение перед едой, делает фото тарелки. Приложение отправляет фото на сервер. YOLOv8 распознает "гречневая каша", вычисляет объем (например, 200 см³). База данных подтягивает плотность гречки (около 0.85 г/см³) и БЖУ. Приложение показывает результат: ~170 грамм гречки, 158 ккал, БЖУ. Пользователь нажимает "Сохранить", запись попадает в локальную историю.

*Сценарий 2. Просмотр истории.*
Пользователь заходит в приложение вечером, открывает вкладку "История" (которая хранится только на устройстве) и видит список всех сфотографированных за день приемов пищи с суммарным подсчетом калорий. 

## 5. Границы MVP

**В MVP входит:**
- Android-приложение (формат `.apk`);
- Добавление функционала ручной коррекции (пользователь может поправить вес, если весы под рукой, или изменить класс продукта, если нейросеть ошиблась);
- Интерфейс камеры и галереи для загрузки фото;
- Локальная база данных (SQLite/Room) на устройстве для хранения истории запросов;
- Серверная часть (API) с развернутой моделью YOLOv8 (обученной на ограниченном датасете базовых продуктов);
- Алгоритм (на сервере), который на основе bounding box и/или масок YOLOv8 выдает: тип продукта и его примерный объем в см³;
- Серверная БД продуктов с полями: название, БЖУ на 100г, калорийность на 100г, примерная плотность (г/см³);
- Вывод итоговых значений (БЖУ, ккал, примерный вес) на экран приложения;
- Доступ мобильного приложения к локальному серверу разработки через **Ngrok** или **Cloudflare Tunnel**.

**Сознательно не входит в MVP:**
- iOS-версия приложения;
- Система авторизации и регистрации (не реализуется из-за сложности и малых сроков выполнения);
- Облачная синхронизация истории и аккаунты пользователей;
- Распознавание сложных составных блюд (супы, салаты из 10 ингредиентов) с высокой точностью;
- Точный расчет объема с использованием лидаров/датчиков глубины (в MVP используется аппроксимация по 2D-фото);
- Ручной поиск и добавление продуктов без фото;
- Сканер штрих-кодов;
- Интеграция с фитнес-трекерами.

## 6. Входные данные

**Основные входы (от клиента к серверу):**
- Изображение блюда (формат `.jpg` или `.png`);
- Опционально: калибровочные данные (например, пользователь указал, что фото сделано со стандартного расстояния, или в кадре есть референсный объект типа монеты, если это реализуемо в рамках MVP для объема).

**Серверная база продуктов:**
- `product_id`
- `name` (совпадающее с классами YOLOv8)
- `density` (г/см³)
- `calories_per_100g`
- `proteins`, `fats`, `carbs`

## 7. Выходные данные и артефакты

**Пользовательский результат (в приложении):**
- Карточка результата с распознанным продуктом;
- Расчетный вес (в граммах);
- Итоговая калорийность и БЖУ блюда;
- Запись в локальной истории устройства с миниатюрой фото и датой.

**Машинно-читаемый результат (JSON ответ от сервера):**
```json
{
  "status": "success",
  "detections": [
    {
      "class": "apple",
      "confidence": 0.92,
      "estimated_volume_cm3": 150.5,
      "estimated_weight_g": 120.4,
      "nutrients": {
        "calories": 62,
        "proteins": 0.3,
        "fats": 0.4,
        "carbs": 13.8
      }
    }
  ]
}
```

## 8. Интеграции и соседние модули

В MVP внешние интеграции минимальны:
- **Единая точка входа (Go Backend):** Все данные от мобильного клиента (фотографии, запросы) проходят исключительно через Go-бэкенд. Клиент не имеет прямого доступа к базе данных или ML-модели.
- **Связь** осуществляется через публичный URL, сгенерированный `Ngrok` или `Cloudflare Tunnel`, который проксирует запросы к Go-бэкенду.
- **Внутренняя интеграция на сервере:** Go-бэкенд является единственным компонентом, имеющим доступ к серверной БД продуктов. Получив фото, бэкенд вызывает инференс YOLOv8, получает классы и размеры, затем делает запрос в БД за плотностью и БЖУ, производит расчет и возвращает клиенту готовый JSON.

## 9. Стек технологий

**Минимальный стек для MVP:**
- **Мобильное приложение:** Kotlin (Android);
- **Бэкенд:** Go (Golang) с использованием фреймворков (например, Fiber) или net/http;
- **ML / Computer Vision:** Экспорт YOLOv8 в ONNX и запуск через Go (onnxruntime), либо выделение ML в отдельный Python-микросервис;
- **Серверная БД:** SQLite или PostgreSQL (хранение справочника продуктов);
- **Локальная БД приложения:** SQLite (Room для Android);
- **Туннелирование:** Ngrok или Cloudflare Tunnel (`cloudflared`);
- **Развертывание сервера:** Docker, docker-compose (для изоляции API и модели).

## 10. Возможные аналоги

- **Calorie Mama / Lose It! (premium)** — используют распознавание фото, но зачастую требуют ручного ввода порции. Наше отличие: фокус на полностью автоматическом расчете веса через объем и плотность в рамках MVP без регистраций.
- **FatSecret / MyFitnessPal** — классические дневники. Отличие: отсутствие рутинного ручного поиска и ввода граммовок.

## 11. Ожидаемый результат

После завершения MVP:
- У пользователя есть установленное `.apk` приложение.
- Приложение умеет делать снимок, отправлять его через туннель на бэкенд и мгновенно (за 2-3 секунды) получать ответ.
- Приложение показывает БЖУ и сохраняет запрос в локальную историю на телефоне.
- Разработчик может легко запускать серверную часть локально одной командой и пробрасывать её в сеть через Ngrok/Cloudflare.

## 12. Полевые испытания

**Испытание:** Использование приложения группой из 3-5 тестировщиков в течение 3 дней для реальных приемов пищи.

**Сценарий:**
- Участники получают `.apk` и актуальную ссылку Ngrok/Cloudflare (вводится в настройки приложения).
- Перед каждым приемом пищи участники фотографируют еду.
- Раз в день участники параллельно взвешивают еду на кухонных весах, чтобы сравнить расчетный вес (от нейросети) с реальным.

**Наблюдаемые данные:**
- Частота падения приложения (крашей);
- Погрешность в определении веса (разница между реальными весами и расчетом объема * плотности);
- Скорость ответа от сервера;
- Удобство просмотра истории.

## 13. Метрики успеха

**Количественные метрики:**
- Точность классификации продукта: >80% (на ограниченном датасете MVP);
- Ошибка оценки веса (через объем и плотность): не более 30% для продуктов простой формы;
- Время обработки запроса (от отправки фото до отображения БЖУ): < 5 секунд;
- Количество успешных сохранений в локальную историю.

**Качественные метрики:**
- Пользователи отмечают, что фотографировать еду быстрее и приятнее, чем вводить данные вручную.

**Признаки провала гипотезы:**
- Оценка объема по одному 2D-фото имеет погрешность >50%, из-за чего калории считаются совершенно неверно.

## 15. Установка, запуск и автоматизация

**Серверная часть:**
1. Запуск бэкенда: `docker-compose up -d` (поднимает Go-бэкенд и БД продуктов).
2. Запуск туннеля: выполнение скрипта `start_ngrok.sh` или запуск cloudflared демона, который проксирует порт бэкенда (например, 8000) в интернет.
3. Копирование полученного публичного URL туннеля.

**Клиентская часть:**
1. Ввод URL туннеля в конфигурационный файл приложения или прямо в интерфейс настроек MVP.
2. Сборка: сборка `.apk` в Android Studio.
3. Установка полученного `.apk` на Android-устройство.

## 16. Безопасность, приватность и права доступа

- **Отсутствие PII (Personal Identifiable Information):** В приложении нет регистрации, сервер не собирает email, имена или пароли. Запросы анонимны.
- **Приватность фото:** Фотографии отправляются на сервер только для инференса и (в рамках MVP) не сохраняются на диске сервера после обработки, чтобы не засорять память и не нарушать приватность.
- **Хранение данных:** Вся история, метрики и потребленные калории хранятся исключительно в локальной базе данных на телефоне пользователя. При удалении приложения данные удаляются.
- **Доступ к серверу:** В рамках учебной практики можно обойтись без сложной защиты, так как URL туннеля генерируется временно.
- **Изоляция базы данных:** Мобильное приложение не имеет прямого подключения к серверной базе данных продуктов. Все запросы к БД выполняются исключительно на стороне Go-бэкенда, что полностью скрывает структуру данных от клиента.

## 17. Риски и ограничения

- **Технический риск:** Оценка 3D-объема по одному плоскому 2D-фото — сложная задача. Без референсного объекта (монета, карточка) или датчика LiDAR погрешность может быть высокой. В MVP это решается принятием средней высоты блюда для определенных классов или использованием bounding box относительно размера кадра.
- **Риск инфраструктуры:** Бесплатные туннели Ngrok меняют URL при каждом перезапуске. Придется либо пересобирать `.apk`, либо делать экран настроек в приложении для ввода нового адреса сервера. Cloudflare Tunnel решает эту проблему, позволяя привязать постоянный домен.
- **Ограниченность датасета:** YOLOv8 будет распознавать только те продукты, на которых обучена. Сложные "мешанины" (рагу, салаты) модель может определить некорректно или определить только один ингредиент.

## 18. Будущая декомпозиция

Задачи для дальнейшего развития (не в MVP):
- Интеграция аутентификации (Firebase Auth) для облачной синхронизации.
- Глобальная база продуктов (интеграция с FatSecret API).
